# テスト計画書

- 版: 第1版（要件定義書 第2版の 9章、CLAUDE.md 7章「テスト規律」に対応)
- 位置づけ: E2Eテスト実装と9月受け入れテストの基準。テストIDは実装するE2Eのファイル名・テスト名に対応させる

## 1. テスト方針（再掲・正はCLAUDE.md 7章）

- 機能コードとPlaywright E2Eは同一コミット。E2Eなしのコミット禁止
- 各機能で4パス必須: **正常系 / 入力エラー系 / 権限系 / 境界値**
- `waitForTimeout` 禁止、`expect(...).toBeVisible()` の自動待機を使う
- セレクタ優先順: `getByRole` > `getByLabel` > `getByText` > `getByTestId`
- バグ修正は再現E2Eを先に書きFAIL確認→修正→ `e2e/regression/` に日付付きで保存
- テストデータはすべて架空値（実受講生の個人情報を使わない）

## 2. テスト環境

| 環境 | 用途 | 時期 |
|---|---|---|
| ローカル（開発機） | E2E開発・実行 | 常時 |
| ステージング（クラウド） | 結合・デモ・低帯域試験 | 7月中〜 |
| 実機（教室16台） | 4デバイス表示確認・負荷試験・受け入れ | 9月（機材搬入後、未決#7） |

テストロール用アカウント（架空値・全環境共通で用意）: 管理者1 / 講師2（担当クラス違い）/ 受講生対面2 / 受講生eL2 / ゲスト1

## 4. 非機能テスト（受け入れ前に実施）

| 試験 | 内容 | 合格基準 |
|---|---|---|
| 16台負荷試験 | 実機16台で90分の演習シナリオ（ログイン→教材→AI質問→Jupyter→提出）を完走。実機搬入前に16クライアント模擬負荷をステージングで1回実施 | システム起因の中断0件（KPI#1）。画面遷移3秒/提出5秒以内（90%ile） |
| 低帯域試験 | 10Mbps帯域制限（Playwright/ブラウザのネットワークスロットリング）でeL受講一巡 | 受講・提出が成立。生徒画面の初回表示1MB以下 |
| 週次バッチ | 月曜7:00スケジュール実行の確認（前倒し実行含む） | 手作業ゼロでレポート閲覧可（KPI#2） |
| リストア演習 | 日次バックアップからステージングを復旧 | 手順書（F6-3）のみで完了 |
| eL自走試験 | 被験者3名（子供・社会人・シニア各1名、架空アカウント） | 問い合わせなしで15分以内に初回提出（KPI#3） |

## 6. 受け入れテストの流れ（9月）

1. E2E全件PASS（ステージング）→ 実機搬入
2. 4デバイス表示確認（全画面）
3. 16台負荷試験（KPI#1）
4. セキュリティ検証（SEC-1〜4＋脆弱性スキャン: Critical/High 0件）
5. リストア演習・週次バッチ確認
6. eL自走試験（KPI#3）
7. 講師トレーニング→講師が自力で1コマ運営（F6判定）

全項目PASSで受け入れ完了。FAILは修正→該当項目のみ再試験（回帰はE2E全件で担保）。
