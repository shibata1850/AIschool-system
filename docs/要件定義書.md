## 1. 概要

### 1.1 目的

2026年10月の開校時点で、対面15名＋eラーニング受講生の「受講管理・演習・到達度管理」をCanvas LMS上で一元化する。差別化の核である3機能（AI講師、プロンプト演習、到達度の自動管理）をCanvas本体を改変せずカスタム層として実装する。

### 1.2 対象業務

受講生管理、教材配信、演習実施（Jupyter・プロンプト演習）、出席管理、到達度の自動集計と可視化、講師の授業運営支援。

### 1.3 期待効果（受け入れ時に測定する完成判定KPI 3点）

新設スクールのためBefore実績が存在しない。よって「導入後にこの数値を満たすこと」を完成判定条件として定義する。

| # | 指標 | 完成判定条件 | 測定方法 |
|---|---|---|---|
| 1 | 授業の安定稼働 | 対面16席同時演習で、90分1コマをシステム起因の中断0件で完走 | 受け入れテスト時に実機16台で実測 |
| 2 | 到達度集計の自動化 | 講師の手集計工数 0時間/週（週次到達度レポートをシステムが自動生成） | 講師業務の実測（開校後1ヶ月） |
| 3 | eラーニング自走率 | 受講登録から初回課題提出まで、受講生が問い合わせなしで15分以内に到達 | 受け入れテストで被験者3名（子供・社会人・シニア各1名）実測 |

### 1.4 用語

| 用語 | 意味 |
|---|---|
| カスタム層 | Canvas本体を改変せず、LTI 1.3外部ツールとREST API連携で実装する独自機能群 |
| LTI 1.3 | LMSと外部ツールを安全に接続する標準規格（Learning Tools Interoperability） |
| 到達度スコア | 課題スコアと出席状況から自動算出する習得状況の指標（算出式は 6.3） |
| 対面 / eL | 教室での対面受講 / eラーニング（自宅）受講の区分 |

```

【4デバイス表示構成】

- Meta Quest 3 ×16（L40SからCloudXR配信）: VR/MR没入授業

  【仮】役割 = GOOVIS不調時の予備＋講師巡回時のサブ画面（常設サブ画面としない）→ 未決事項#4

【eラーニング】
- 受講生の自宅回線（品質不明）。低帯域（実効10Mbps想定）でも受講可能なこと
- 対応デバイス: PC（Chrome/Edge/Safari 最新2バージョン）、iPad（Safari）、スマホ（閲覧・提出のみ）

### 2.2 利用者制約

```
- 受講生: 小中高生 / 社会人 / シニア。ITリテラシーの幅が極端に広い

- HMD連続装着時間の上限を1コマの授業設計に組み込む
  【仮】連続30分装着ごとに10分の非HMD活動を挟む（実機検証で確定 → 未決事項#9）
- シニア: 文字サイズ大、1画面1タスク、専門用語に平易な言い換え併記
- 講師: SOFTDOING社員（AIツール研修済み）。専任情報システム担当はいない前提で、
  運用手順は「誰でもできる形」で文書化する
```

### 2.3 既存システム連携

```
- 会計・請求: フェーズ1では連携しない（既存手段で運用継続）【要確認: 使用ソフト】

  SSOはフェーズ2で検討）。ID・初期パスワードは入会手続き時に発行

内装工事完了と機材搬入は9月見込み（未決事項#7で確定）。実機16台での結合テストは搬入後にしかできないため、**8月末までにステージング環境で機能を完成させておく**。

## 4. 機能仕様（5W1H＋入力項目＋例外）

### F1: Canvas LMS基盤

- Who: 全ユーザー（管理者・講師・受講生）
- When: 常時（授業時間帯 平日夕方〜夜・土日を高負荷帯として設計）
- Where: 教室のNUC/GOOVIS、eラーニングの自宅端末
- What: コース管理・教材配信・課題・出席の基盤
- Why: 学習運営の一元化。個別ツールの寄せ集め運用を排除する
- How: Docker（Rails+PostgreSQL+Redis）でクラウドにセルフホスト。校内L40SとはLTI/APIで接続
  - ホスティング先: 【仮】さくらのクラウド（石狩リージョン）→ 未決事項#2
    根拠: 未成年の学習データを国内法準拠のデータセンターに置く。円建てで補助金実績報告の費用説明が単純。AWS比で月額固定費が読みやすい
  - 構成: 【仮】アプリ4vCPU/8GB ×2台（冗長）、PostgreSQL 4vCPU/16GB、Redis 2GB、オブジェクトストレージ（教材・提出物）

初期セットアップで管理者が登録する項目:

| 項目 | 型 | 必須 | 制約・例 |
|---|---|---|---|
| コース名 | 文字列 | ○ | 最大100文字。例:「プロンプトエンジニアリング基礎（2026秋・対面）」 |
| クラス区分 | 選択 | ○ | 対面 / eL のいずれか |
| 開講期間 | 日付範囲 | ○ | 例: 2026-10-05 〜 2027-03-19 |
| 授業コマ | 日時＋90分 | ○ | 週次繰り返し登録可。例: 毎週火曜 17:00-18:30 |
| 受講生登録 | CSV取込 or 個別 | ○ | 列: 表示名（ニックネーム可）, メール, クラス区分, 年齢区分（小中高/社会人/シニア） |

例外パターン:
1. クラウド障害時 → 教室授業はL40Sローカル資材（配布済み教材・Jupyter）で続行できる授業運営手順を用意（F6ドキュメントに含める）
2. 校内LAN障害時 → 講師機のモバイル回線で最低限の出席記録を取れる代替手順
3. 同時接続がeラーニング側で急増した場合 → 想定同時接続数（【仮】50、設計上限100 → 未決事項#8）の2倍まで性能劣化なし、超過時は静的教材閲覧のみ維持
4. 授業中のCanvasアップデート禁止 → メンテナンス窓は日曜 5:00-7:00 に固定。窓外の緊急パッチは管理者承認＋授業スケジュール確認の上でのみ実施

### F2: AI講師・Jupyter連携

- Who: 受講生（演習）、講師（監督・介入）
- When: 対面授業中およびeラーニング自習時
- Where: 対面はGOOVIS/Quest 3、eラーニングはブラウザ
- What: 受講生の質問にAIが応答し、Jupyter演習の実行環境を提供する
- Why: 講師1名で15名の個別対応を成立させる（人間は補助に特化する設計）
- How: L40S上のJupyterHub（16名同時）。AI応答は抽象化レイヤー経由
  - 推論基盤: 【仮】フェーズ1はClaude API（個人情報マスキング必須・5.3準拠）、抽象化レイヤーによりL40SローカルLLMへ切替可能に実装 → 未決事項#3
    根拠: 応答品質と開校までの実装工数を優先。L40SはCloudXR配信・JupyterHubの負荷を優先的に割り当てる
  - LTI 1.3ツールとしてCanvasコース内に埋め込み。JupyterHub認証はLTIログインと連動（再ログイン不要）

AI講師チャットの入力項目:

| 項目 | 型 | 必須 | 制約・例 |
|---|---|---|---|
| 質問文 | テキスト | ○ | 最大2,000文字。例:「forぶんとwhileぶんのちがいを教えて」 |
| 質問カテゴリ | 選択 | - | 課題について / 操作方法 / その他（未選択時は「その他」） |
| 対象課題 | 自動付与 | - | 演習画面から起動した場合、課題IDを自動で紐付け |

AI応答の表示要件:
- 応答は必ず未成年向けコンテンツフィルタを通過後に表示（フィルタ未実装のまま直接表示する構成は禁止）
- 回答には「AIによる回答です。わからないときは先生に聞いてね」の定型注記を常時表示
- 専門用語には平易な言い換えを添える（システムプロンプトで指示）

例外パターン:
1. AI応答が5秒を超える場合 → 「考え中」表示と定型ヒントのフォールバック（10秒でタイムアウト、再試行ボタン表示）
2. AIが不適切応答を返す可能性 → 未成年向けコンテンツフィルタを必ず経由。フィルタでブロックされた場合は「この質問にはお答えできません。先生に聞いてください」を表示し、講師へ通知
3. 推論基盤（L40SまたはAPI）停止時 → 演習は静的教材モードに自動切替、講師へ即時通知（講師ダッシュボードに赤色バナー＋音）
4. 受講生がAIに個人情報を入力した場合 → ログ保存前にマスキング（氏名・電話番号・住所・メールアドレスのパターン検出。検出時は受講生にも「個人情報は入力しないでね」と注意表示）
5. JupyterHubのカーネルが応答しない場合 → 受講生画面に「再起動」ボタンを表示（ワンタップで自分のカーネルのみ再起動、他の受講生に影響しない）

### F3: プロンプト演習管理

- Who: 受講生（提出）、講師（確認・差戻し）、AI（一次採点）
- When: 授業内演習・宿題
- Where: 4デバイスすべて＋eラーニング
- What: プロンプト課題の出題・提出・AI一次採点・講師確認
- Why: プロンプト力の定量評価を差別化商品にする
- How: LTI 1.3外部ツールとして実装。採点結果はCanvasの成績表へAPI連携

出題時の入力項目（講師）:

| 項目 | 型 | 必須 | 制約・例 |
|---|---|---|---|
| 課題タイトル | 文字列 | ○ | 最大100文字。例:「お店の紹介文をAIに書かせよう」 |
| 課題説明 | リッチテキスト | ○ | 最大10,000文字。画像添付可（1枚5MBまで） |
| 提出期限 | 日時 | ○ | 例: 2026-10-13 23:59 |
| プロンプト文字数上限 | 整数 | ○ | 100〜8,000。既定値4,000 |
| 採点観点 | 観点リスト | ○ | 各観点 = 名称（最大30文字）＋配点（合計100）。既定テンプレート:「指示の明確さ40 / 条件指定30 / 出力形式の指定30」 |
| 採点プロンプト | テキスト | ○ | AI一次採点に使う指示文。テンプレートから選択可 |

提出時の入力項目（受講生）:

| 項目 | 型 | 必須 | 制約・例 |
|---|---|---|---|
| プロンプト本文 | テキスト | ○ | 課題の文字数上限まで。例:「あなたはパン屋の店長です。新商品のメロンパンを…」 |
| AI実行結果の貼り付け | テキスト | - | 最大16,000文字 |
| ふりかえりメモ | テキスト | - | 最大500文字。例:「条件を3つに増やしたら思い通りになった」 |

AI一次採点の出力項目:

| 項目 | 型 | 説明 |
|---|---|---|
| 総合スコア | 整数 0-100 | 観点別スコアの合計 |
| 観点別スコア | 整数の配列 | 出題時の採点観点ごと |
| 講評 | テキスト | 受講生向け。ほめる点1つ以上＋改善点1つ、平易な日本語 |
| 採点根拠 | テキスト | 講師向け（受講生には非表示） |

状態遷移:

```
[未着手] →開始→ [取組中] →提出→ [提出済] →AI採点→ [AI採点済]
                                                │
                                          講師確認
                                          ├→ [完了]（成績表へ反映）
                                          └→ [差戻し] →修正→ [取組中]
```

- 状態を変更できるのは: 開始・提出・修正=受講生本人、講師確認（完了/差戻し）=講師・管理者、AI採点=システムのみ
- 差戻し時、講師はコメント（最大1,000文字）必須

例外パターン:
1. 同一課題の重複提出 → 最新版のみ有効、旧版は履歴保持（受講生・講師とも履歴閲覧可、版数表示）
2. AI採点と講師採点が乖離した場合 → 講師採点を最終とし、乖離ログを蓄積（採点プロンプト改善に使用）。乖離=総合スコア差20点以上を自動フラグ
3. 提出期限後の提出 → 受付するが「遅延」フラグ、講師承認で成績反映
4. 文字数上限超過 → 提出前にクライアント側で検知しエラー表示（残り文字数を入力中に常時表示）
5. AI採点が失敗（タイムアウト・エラー）した場合 → [提出済]のまま保持し、10分間隔で最大3回自動リトライ。3回失敗で講師へ通知し手動採点に切替
6. 提出直後の受講生による取消 → 提出後5分以内は「取り消して修正」可（AI採点開始前のみ）

### F4: 学習ログ・到達度ダッシュボード

- Who: 講師（授業中モニタリング）、管理者（週次レポート）
- When: 授業中リアルタイム＋週次自動集計
- Where: 講師機（NearHub S65投影）、管理者PC
- What: 出席・演習進捗・提出状況・到達度スコアの可視化。「知識・技能」の習得状況を自動判定
- Why: 完成判定KPI#2（講師の手集計0時間）の実現
- How: Canvas REST APIからログ収集（【仮】5分間隔ポーリング＋授業中は1分間隔）→独自集計→ダッシュボード表示（Next.js）

授業中モニタリング画面の表示項目（受講生1名につき1タイル、16タイル格子）:

| 項目 | 表示 |
|---|---|
| 表示名・座席番号 | タイル上部に常時表示（氏名フルネームは表示しない） |
| 現在の課題状態 | 未着手（灰）/ 取組中（青）/ 提出済（緑）/ 差戻し（橙）の色分け |
| 最終操作からの経過時間 | 5分以上操作なしで「停滞?」バッジ表示 |
| AIへの質問回数 | 当該コマ内の回数 |
| ヘルプ要請 | 受講生が「先生を呼ぶ」ボタンを押すとタイルが点滅（NearHubタッチで消込） |

週次到達度レポートの内容:
- 生成タイミング: 毎週月曜 7:00 に自動生成、講師・管理者へCanvasメッセージで通知
- 単位: クラス別＋受講生別
- 内容: 出席率 / 課題提出率 / 到達度スコア推移（週次） / 停滞アラート（2週連続で到達度が下降した受講生の一覧） / 未提出課題一覧

例外パターン:
1. 途中入会者 → 未受講回は到達度計算の分母から除外
2. 退会者 → ダッシュボード非表示、データは保持期間ルール（5.3）に従い削除
3. データ欠損コマ（システム障害等） → 「計測不能」表示とし、0点扱いにしない
4. 出席したが演習未提出 → 「出席・未提出」を区別して表示
5. Canvas API のレート制限に達した場合 → 収集間隔を自動で2倍に延伸し、ダッシュボードに「データ更新遅延中（最終更新 hh:mm）」を表示

### F5: テスト・品質保証・セキュリティ検証

- Who: 開発者（ノーティ/SOFTDOING、分担は未決事項#1）
- When: 各機能の実装と同時（E2E同一コミット原則）＋受け入れ前の総合検証
- What: Playwright E2E（正常系・入力エラー系・権限系・境界値の4パス）、16台負荷試験、低帯域試験、セキュリティ検証
- How: CLAUDE.md 7章「テスト規律」に従う。セキュリティ検証の範囲:
  - 権限外URL直接アクセスの403確認（全ロール×全画面の総当たり）
  - LTI 1.3署名検証・リプレイ攻撃対策の確認
  - 監査ログの追記専用性（アプリDBユーザーにUPDATE/DELETE権限がないこと）の確認
  - 個人情報マスキング（F2例外4）の動作確認
  - 依存パッケージの既知脆弱性スキャン（受け入れ前に1回＋月次）

### F6: 運用ドキュメント・操作指導

- Who: 講師・管理者（読者）、開発者（作成）
- What: 「誰でもできる形」の運用手順書＋操作指導（講師トレーニング）
- 成果物一覧:
  1. 授業運営手順書（通常時の1コマの流れ: 機材起動→出席→演習→終了）
  2. 障害時運転手順書（F1例外1・2、F2例外3の代替運用。判断フローチャート付き）
  3. 週次・月次運用チェックリスト（バックアップ確認、脆弱性スキャン、メンテナンス窓の作業）
  4. 受講生向けはじめてガイド（対面用・eL用の2種。eL用はKPI#3「15分で初回提出」を満たす構成）
  5. 個人情報対応手順（開示・削除請求への対応。5.3準拠）
- 操作指導: 9月中下旬に講師向けトレーニングを実施し、「講師が自力で1コマ運営できる」ことを完了条件とする

### 5.1 権限マトリクス

| 他人の到達度閲覧 | ⭕ | ⭕（担当クラスのみ） | - | - | - |
| 採点・差戻し | ⭕ | ⭕ | - | - | - |
| 授業中モニタリング閲覧 | ⭕ | ⭕（担当クラスのみ） | - | - | - |

- ゲストロールは体験会用に閲覧専用コースのみ付与可（フェーズ1では管理者が手動発行）

### 5.2 監査ログ要件

- 対象: 全データの作成・更新・削除（操作者・日時・変更前後の値）
- 閲覧権限: 管理者のみ
- 保持期間: 【仮】在籍中＋卒業（退会）後3年 → 未決事項#10（Pマーク運用と整合確認のうえ確定）
- 改ざん防止: ログテーブルはアプリケーションから追記のみ（UPDATE/DELETE権限なし）

### 5.3 個人情報の取り扱い

- 未成年の学習データを含むため要配慮データとして扱う
- 保護者からの開示・削除請求への対応手順を運用ドキュメント（F6-5）に含める
- AI推論に外部API（Claude API等）を使う場合、送信データから氏名・連絡先を除去する（F2例外4のマスキングを送信前に適用）
- 学習ログ・提出物の保持期間: 監査ログと同じく【仮】在籍中＋退会後3年、退会後の削除はバッチで月次実行し監査ログに記録
- 画面・帳票・ログでの氏名表示は原則「表示名（ニックネーム可）」とし、氏名フルネームは管理者のユーザー管理画面のみ

## 6. データモデル

Canvas標準エンティティ（User / Course / Assignment / Submission / Enrollment）を正とし、独自エンティティは以下に絞る。独自エンティティはカスタム層DB（PostgreSQL、Canvas本体DBとは分離）に置き、CanvasとはIDで参照する。

### 6.1 独自エンティティ

**受講生プロフィール（student_profiles）** — Canvas Userの拡張

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|
| canvas_user_id | 整数 | ○ | Canvas UserのID（一意） |
| 区分 | 列挙 | ○ | 対面 / eL |
| 年齢区分 | 列挙 | ○ | 小中高 / 社会人 / シニア |
| 未成年フラグ | 真偽 | ○ | コンテンツフィルタ・VR可否の判定に使用 |
| 入会日 / 退会日 | 日付 | ○/- | 退会日は退会時に設定（到達度の分母制御・削除バッチに使用） |

**デバイス割当（device_assignments）**

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|
| 座席番号 | 整数 1-16 | ○ | 教室の物理座席 |
| NUC識別子 / GOOVIS識別子 / Quest3識別子 | 文字列 | ○ | 資産管理番号（シリアルは別台帳） |
| canvas_user_id | 整数 | - | 当日割当の受講生。授業コマ単位で付け替え可 |
| 割当日 | 日付 | ○ | |

**AI採点結果（ai_grading_results）**

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|
| canvas_submission_id | 整数 | ○ | Canvas SubmissionのID |
| 版数 | 整数 | ○ | 同一課題の再提出を区別 |
| 総合スコア / 観点別スコア | 整数 / JSON | ○ | F3仕様に対応 |
| 講評 / 採点根拠 | テキスト | ○ | 講評=受講生向け、根拠=講師向け |

| 乖離フラグ | 真偽 | ○ | 既定false |

**到達度スコア（achievement_scores）**

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|
| canvas_user_id / canvas_course_id | 整数 | ○ | |
| 集計週 | 日付（週の月曜） | ○ | |
| 出席率 / 提出率 / 課題平均スコア | 数値 0-100 | ○ | 分母は在籍期間内のコマ・課題のみ |
| 到達度スコア | 数値 0-100 | ○ | 6.3の算出式 |
| 計測不能フラグ | 真偽 | ○ | データ欠損週はtrue（0点扱いにしない） |

**出席記録（attendance_records）**

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|

| 記録方法 | 列挙 | ○ | 自動（ログイン検知）/ 講師手動 / 代替手順（LAN障害時） |

**監査ログ（audit_logs）** — 追記専用

| 列 | 型 | 必須 | 説明 |
|---|---|---|---|
| 操作者ID / 操作者ロール | 整数 / 列挙 | ○ | |
| 操作種別 | 列挙 | ○ | 作成 / 更新 / 削除 / 閲覧（監査ログ自身の閲覧も記録） |
| 対象エンティティ / 対象ID | 文字列 / 整数 | ○ | |
| 変更前 / 変更後 | JSON | -/- | 作成時は変更前なし、削除時は変更後なし |
| 日時 | タイムスタンプ | ○ | |

### 6.2 Canvas標準に載せるもの（独自DBに持たない）

- 教材・課題文・提出本文・成績表 → Canvas標準（Assignment / Submission / Gradebook）
- コース・クラス編成 → Canvas Course + Section（対面/eLはSectionで区分）

### 6.3 到達度スコア算出式（単体テスト必須）

```
到達度スコア = 課題平均スコア × 0.6 + 提出率 × 0.2 + 出席率 × 0.2
```

- 【仮】重み（0.6 / 0.2 / 0.2）は開校後1ヶ月の実データで見直す前提。設定値として外部化し、変更は管理者のみ・監査ログ記録
- 分母規則: 途中入会者は入会日以降のコマ・課題のみ。「計測不能」週は分母から除外
- 端数: 小数第1位まで、四捨五入

## 9. 受け入れ基準

### 9.1 テスト方針

- Playwright E2E必須（正常系・入力エラー系・権限系・境界値の4パス）
- 全ロール（管理者/講師/受講生対面/受講生eL）でのログイン・権限確認
- 4デバイス表示確認: GOOVIS実機 / Quest 3実機 / NearHubタッチ / モバイルモニター
- 負荷: 実機16台同時演習で90分完走（KPI#1）
- 低帯域: 10Mbps回線シミュレーションでeラーニング受講が成立すること
- テストデータはすべて架空値（実受講生の個人情報を使わない）

### 9.2 機能別完成判定

| 機能 | 完成判定条件 |
|---|---|
| F1 | ①ステージングURLで4ロール全員がログインでき、権限マトリクス通りに画面が制限される ②日曜メンテ窓での更新リハーサル1回成功 ③バックアップからのリストア演習1回成功 |
| F2 | ①受講生の質問→フィルタ経由→応答表示が5秒以内（10問中9問以上） ②推論基盤停止を模擬し、静的教材モードへの自動切替と講師通知が動作 ③個人情報を含む質問でマスキングが動作し、外部APIへ原文が送信されない ④JupyterHubに16名同時ログインし全員のカーネルが起動 |
