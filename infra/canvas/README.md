# Canvas LMS（OSS版）構築資材

- 位置づけ: さくらのクラウド上にCanvas LMS本体を構築するための資材一式。
  **Canvas本体のソースコードは本リポジトリに置かない**（CLAUDE.md 絶対ルール1）。
  サーバーが公式リポジトリ（instructure/canvas-lms）を直接クローンして構築する
- 手順の正: `docs/Canvasステージング構築手順.md`。本ディレクトリはその実行部
- 調査根拠: `docs/Canvas調査メモ.md`（保守された公式本番Dockerイメージは存在
  しないため、公式リポジトリ同梱のDocker構成を使う）

## 構成

| ファイル | 用途 |
|---|---|
| `setup-staging.sh` | さくらのサーバー（Ubuntu）で実行する構築スクリプト。公式リポジトリを指定バージョンでクローンし、同梱のDockerセットアップへ誘導する |
| `.env.staging.example` | カスタム層からステージングCanvasへ接続するための環境変数の見本 |

## 使い方（さくらのクラウドのサーバー確保後）

1. サーバー（Ubuntu 24.04 LTS想定、4vCPU/8GB/50GB以上）にSSHログイン
2. 本リポジトリの `infra/canvas/` をサーバーへコピーして実行:

```
bash setup-staging.sh
```

3. スクリプトは次を行う:
   - Docker / Docker Compose / git の存在確認（なければ導入コマンドを案内）
   - 公式リポジトリを **バージョン固定で浅くクローン**（既定はスクリプト内の
     `CANVAS_REF`。更新時はここだけ変える）
   - クローン後、Canvas同梱の `doc/docker/developing_with_docker.md` に沿った
     セットアップ（`./script/docker_dev_setup.sh`）へ誘導
4. 初回セットアップで作る管理者アカウントは実運用値（パスワード管理台帳へ）。
   **受講生の実個人情報はステージングに投入しない**（架空値のみ）
5. 構築後、`docs/Canvasステージング構築手順.md` 3章（トークン発行→カスタム層の
   `.env` 設定→接続確認）へ進む

## バージョン更新の手順

1. `setup-staging.sh` の `CANVAS_REF` を新しい安定リリースに変更してコミット
   （何をいつどの版に上げたかがgit履歴に残る）
2. サーバー側で再実行（メンテナンス窓: 日曜5:00-7:00 — 要件定義書F1例外4）

## やってはいけないこと

- Canvasのソースコードを本リポジトリ（または自分たちのGitHub）へコピー・フォーク
  すること。セキュリティ更新の追従責任を自ら抱え込むことになり、CLAUDE.md
  絶対ルール1にも違反する
- クローンしたCanvasソースをサーバー上で直接編集すること（カスタマイズは
  LTI 1.3 外部ツールとREST API経由のみ)
- 本番（10月開校）をこのステージング構成のまま使うこと。本番は公式の
  Production Start ガイドに沿って再構築し、日次バックアップを設定する
