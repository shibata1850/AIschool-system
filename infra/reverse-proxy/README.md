# リバースプロキシ（Basic認証）— カスタム層の仮アクセス制限

- 目的: 公開中のカスタム層（`:3000`）に**パスワード（HTTP Basic認証）**をかける
- 位置づけ: 実データ投入前の**暫定的な関所**。本来の保護は TLS化＋LTI 1.3ログイン
  （`docs/実装状況.md` セキュリティ残課題）。それまでの間の最低限の防御
- 併用推奨: さくらのパケットフィルタで接続元IPも制限する（二重の防御）

## 正直な限界

- いまは平文HTTPのため、Basic認証のパスワードも暗号化されずに流れる（盗聴に弱い）。
  無差別アクセス・ボットは弾けるが、盗聴対策にはならない → TLS化までの暫定策
- 本物のログイン（誰がどの権限か）はLTI 1.3導入まで未対応。Basic認証は
  「入口の共通パスワード」であり、ロール（管理者/講師/生徒）は区別しない

## 手順（さくらのサーバー上・Coworkが実施）

前提: カスタム層コンテナ `aischool-custom` が `:3000` で稼働中。

1. パスワードファイルを作成（`kanri` はユーザー名例。パスワードは対話入力）:

```
sudo apt-get install -y apache2-utils
htpasswd -c ~/proxy-htpasswd kanri
```

（`~/proxy-htpasswd` は**サーバー上のみ**に置く。リポジトリにコミットしない）

2. この設定ファイルを使ってnginxコンテナを起動（`:8080` を公開、`:3000` は
   localhostのみに閉じるのが理想。ポート構成は実環境に合わせて調整）:

```
docker run -d --name aischool-proxy --network host \
  -v ~/AIschool-system/infra/reverse-proxy/nginx.conf.template:/etc/nginx/conf.d/default.conf:ro \
  -v ~/proxy-htpasswd:/etc/nginx/.htpasswd:ro \
  nginx:alpine
```

3. さくらのパケットフィルタで、公開するのはプロキシのポート（例 `8080`）のみにし、
   `3000` への直接アクセスは閉じる（プロキシを迂回されないため）

4. ブラウザで `http://133.125.225.64:8080/` を開き、パスワードを求められることを確認

## パスワードの管理

- ユーザー名・パスワードはパスワード管理台帳へ（チャット・コミット禁止）
- `~/proxy-htpasswd` はサーバー上のファイル。リポジトリには**入れない**
