## 1. プロジェクト概要

- 事業者: SOFTDOING株式会社
- 対象: Next Gen AI School（岩手県北上市・北上駅西口、2026年10月開校）
- 本リポジトリの範囲: Canvas LMS OSS版を基盤とした**カスタム層**の開発
  - LTI 1.3 外部ツール群（AI講師連携、プロンプト演習管理）
  - Canvas REST API 連携（学習ログ収集、到達度ダッシュボード）
  - JupyterHub 連携
- 本リポジトリの範囲外: Canvas LMS 本体のソースコード
- 詳細仕様: `docs/要件定義書.md` を必ず参照してから実装に着手する

## 2. 絶対ルール（違反禁止）

1. **Canvas LMS 本体をフォーク・改変しない。** カスタマイズは LTI 1.3 外部ツールと REST API / GraphQL 経由のみ。本体改変が必要に見えた場合は実装せず、代替設計を提案して人間の判断を仰ぐ
2. **E2Eテストなしのコミット禁止。** 詳細は「7. テスト規律」
3. **受講生の実個人情報（氏名・連絡先・学習ログ）をコード・fixtures・コミットに含めない。** テストデータは架空値のみ
4. **シークレット（APIキー、DB接続情報）をコミットしない。** `.env` は gitignore 済みであることを毎回確認
5. **UIテキストはすべて日本語。** 対象ユーザーは小中高生からシニアまで。専門用語には振り仮名または平易な言い換えを添える
6. 破壊的操作（DB削除、本番デプロイ、force push）は実行前に必ず人間へ確認する

## 7. テスト規律（Definition of Done）

SOFTDOING 社の E2E 義務化ルールを本リポジトリにも適用する。

機能が「完了」となる条件（すべて必須）:

- [ ] 機能コードを書いた
- [ ] 該当機能の Playwright E2E テストを書いた（正常系・入力エラー系・権限系・境界値の4パス）
- [ ] E2E がローカルで PASS した
- [ ] 4デバイス条件（GOOVIS 解像度 / Quest 3 想定 / NearHub タッチ / モバイルモニター）で表示確認した
- [ ] 既存テストが全て通る（リグレッションなし）
- [ ] テストコードを本体コードと同じコミットに含めた

追加ルール:

- バグ修正は「再現E2Eを先に書き、FAILを確認してから直す」。修正後のテストは `e2e/regression/` に日付付きで残す
- `waitForTimeout` の固定待機禁止。`expect(...).toBeVisible()` の自動待機を使う
- セレクタ優先順: `getByRole` > `getByLabel` > `getByText` > `getByTestId`。CSSセレクタは最終手段
- `.skip` `.only` を残したコミット禁止
- 負荷要件: 演習系機能は **16クライアント同時操作** を模した負荷テストを受け入れ前に1回実施

## 8. コーディング規約

- コミットは Conventional Commits 準拠（例: `feat: プロンプト演習の提出機能を追加（E2Eテスト付き）`）
- 機能コードと E2E は同一コミット。E2E だけ後付けするコミット禁止
- 到達度スコア等の計算ロジックには必ず単体テストも付ける（E2E の代替にはならない、両方書く）
- ログ出力に個人情報を含めない（ユーザーIDのみ可、氏名不可）

## 9. セキュリティ・個人情報

- 受講生には未成年が含まれる。学習ログ・提出物は要配慮の個人データとして扱う
- 監査ログ: データの作成・更新・削除はすべて記録（操作者・日時・変更前後）。閲覧は管理者ロールのみ
- AI講師の出力には未成年向けコンテンツフィルタを必ず通す。フィルタ実装前に AI 応答を生徒へ直接表示する機能をマージしない
- SOFTDOING 社の Pマーク運用と整合させる。保持期間・削除手順は要件定義書 5.3 に従う
